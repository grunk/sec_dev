S-SDLC

Secure Software development life cycle

Avant

  • Cycle traditionnel (cascade, V, agile, ...)
  • Sécurité au moment des tests finaux
Le problème : bcp de vulnérabilités trouvées à la fin du projet voir pas trouvée du tt

Définition S-SDLC

  • Méthodologie globale
  • Standard pour le développement
  • Défini les bonnes pratiques
  • Réduit les vulnérabilités
Identique à cycle de dév classique (V , agile) sauf qu'on intègre * Méthodologie globale pour un cycle de dev sure
* Le but est de tendre vers un standard
* Bonne pratique de développement sécurisé en fonction de la phase du projet (design, codage, maintenance) * Doit conduire à une réduction des vulnérabilités

Qu'est ce qu'une vulnérabilité ?

Un défaut dans un logiciel

  • Elevation de privilège
  • Compromission de données
* Je peux voir ou faire quelque chose sans en avoir les droits
* Accès à des données (lecture ou ecriture) par une voie non prévue (outrepasser un paywall par ex)

TOP 3

  1. Buffer overflow (CWE-119)
  2. XSS (CWE-79)
  3. IIV (CWE-20)
1- Principalement dans le langages sans gestion de mémoire (C/C++)
=> Execution de code malicieux , crash, accès à des données non autorisé

2- Pricipalement vol de données (cookie). Execution de code possible si lié à d'autre failles

3- Improper input validation : Crash , accès à des données.

Avantage du S-SDLC

  • Détection/Résolution des vulnérabilités relativement tôt
  • Réduction des coûts
  • Prise de conscience de la sécurité par toutes les parties prenantes
  • Globalement moins de risque pour la société
* On y a pensé en amont, on peut donc les résourdres vite.
* Plus facile = moins cher
* Ne concerne plus que les développeur mais aussi tout l'opérationnel
* Moins de failles = moins de risque de mise en cause pour l'entreprise (ex botnet dahua : mauvaise réput)

Comment ca marche ?

Ajout des notions de sécurité dans un process de développement

Ex : écrire les contraintes sécuritaires en même temps que le besoin fonctionnel

Exercice

Page de login

  • Connexion identifiant / mot de passe
  • Email de confirmation
  • Récupération de mot de passe
  • Connexion "permanante"
Par groupe
* Longueur mdp et identifiant
* Stockage mdp (hash, bcrypt)
* Ne pas renvoyer le mdp par mail
* Durée de validité limité récup mdp
* Attention Contenu cookie connexion permanante
